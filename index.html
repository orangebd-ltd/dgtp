Design Principles

Five principles behind every decision in this design

Every architecture choice in the LLD and HLD was guided by these five principles. They explain the why behind what you will see in the slides that follow.

Open, Standards-Based Integration
Every registry connection uses an open, industry-standard protocol — SOAP, REST, OData v4, or ArcGIS REST. The APIM gateway exposes a standard OpenAPI interface. Azure Entra federates with the national eID via standard OIDC. No proprietary lock-in: if a vendor changes, only the relevant adapter or configuration changes — nothing else.
Authoritative Source, Loosely Coupled
Each registry remains the single source of truth for its own data — nothing is migrated or duplicated permanently. The platform only holds short-lived cached copies that expire automatically. Each adapter is independent: if the Civil Registry goes down, the Business Registry and others continue working. No single failure can bring down the whole platform.
Security, Identity & Privacy by Design
Security is not added on top — it is built into every layer. Every request passes through 8 security gates before reaching a registry. Citizens authenticate with their national eID and can only access their own data. Citizen identities in the audit log are stored as scrambled codes. Data never leaves Sint Maarten. These rules are enforced at the architecture level and cannot be overridden.
Resilience & Once-Only by Default
Citizens provide their information once — the platform fetches it automatically from the relevant registry. No re-entry, no repeated documents. Data updates are delivered reliably through the Outbox pattern — zero data loss even if the system crashes mid-transfer. CMF records stay fresh within 15 minutes. The audit log retains every access for 7 years. Compliance is a built-in guarantee, not an afterthought.
Observable & Reproducible Infrastructure
Every component emits structured logs with a unique tracking ID, so any request can be traced end-to-end across all registries. Alert rules fire automatically when thresholds are breached. All infrastructure is declared as code — a new engineer can reproduce the entire platform from a single pipeline run. No tribal knowledge, no manual configuration. Every change is reviewed, version-controlled, and auditable.

Component 04 — Data Layer

Where the platform stores data — and why

✅
The registries keep their own databases — nothing is migrated
None of the six registries' existing data is moved into Azure SQL or Cosmos DB. The Civil Registry, Business Registry, and others continue running on their own systems exactly as before. The platform only stores what it fetches and translates — a temporary copy in Cosmos DB that expires automatically, and an audit trail of every access in Azure SQL. The registries remain the single source of truth.
🔒 Hard rule: Citizen personal data lives only in Cosmos DB with a short expiry time. Redis holds only technical tokens — no citizen data, ever. The Audit Log stores citizen identity as a scrambled code, never as a readable name.
⏱ Proposed data expiry times How long each registry's data is stored before the platform re-fetches it
⚠ Requires confirmation from each registry owner ▾

Different data changes at different rates. A business status can change within minutes; a home address rarely changes. The expiry time is set to match how often the data is likely to change — so the portal always shows current information without calling the registry more than necessary. Each registry owner needs to confirm whether these expiry times are appropriate for their data.

Outbox — staging area before dispatch
When an adapter saves data, it writes a record here in the same operation. If anything fails after that, this record ensures the message is never lost. Saving the data and recording the intent to send it happen together — or not at all.
Example: Marcus registers his business. At 09:14:32 the adapter writes his business record to Cosmos DB and a row to the Outbox — in one single step. At 09:14:33 the system crashes. At 09:14:45 the system recovers, finds the Outbox row marked pending, and completes the dispatch. Marcus never notices anything went wrong.
Audit Log
kept 7 years · append-only
A permanent record of every data access — who requested what, when, and from which registry. Stored in Azure SQL because it needs enforced 7-year retention and can never be edited or deleted. Citizen identities stored as a scrambled code.
Shared registry code dictionary
Registries store many values as short codes — for example, marital status as ONGEHUWD or a company type as BV. This table translates those codes into plain readable labels in both English and Dutch, so the portal always displays something a citizen can understand rather than a raw code.

Safety net
⚠️
If delivery fails
Failed messages are held in a separate queue. A monitor checks every 15 minutes. If any message has been sitting undelivered for more than 3 days, the operations team is automatically alerted — this is treated as an incident requiring immediate attention.

Every message has a standard wrapper
Each update includes a unique tracking ID, a version number, and the full translated data payload. This means the receiving system always knows exactly what it received, can detect duplicates, and can trace every message back through the full audit trail.
No update is ever lost. No data is shared between registries. Every access is permanently logged.

01

Something failed — and was missed.
The platform checks for undelivered messages every 15 minutes. If any message has been sitting undelivered for more than 3 days, the operations team is automatically alerted. A 4-day delay means that alert either was not set up correctly, or fired and was ignored. The CMF copy should never be more than 15 minutes behind a registry update — a 4-day gap is a serious failure that needs immediate investigation.

Needs confirmation today.
The current design places citizen notifications in Phase 2 — not active at launch. The audit log and CMF refresh both work without it. However, the team needs to confirm this decision: is Phase 2 the right call, or should notifications be part of the initial launch scope? If the decision is Phase 2, that must be explicitly documented — otherwise there is a risk someone adds it early, creating a listener with no consumer that fills the failure queue and triggers false alerts.
Component 06 — Identity

How a citizen logs in — and how the platform uses that login

The platform connects to Sint Maarten's national eID system. Here is what happens from the moment a citizen taps "Login" to the moment their registry data is returned.

💡
Azure Entra External ID is part of this platform — not the eID. The national eID is owned and operated by government, outside this project. Entra External ID is the component this project builds — it acts as a broker that receives the eID confirmation and translates it into a session token the platform's own services can use. Think of it as the platform's front door for logins.
(1)
Citizen logs in via national eID
The portal redirects to the national eID login page. The platform never sees the citizen's password or credentials — authentication is handled entirely by the eID provider.
(2)
eID confirms identity — Entra converts it into a platform token
The eID sends a confirmation back using a standard protocol (OIDC). Azure Entra External ID — which is part of this platform, not the eID — receives this confirmation and converts it into a short-lived session token that the platform's own services understand. The eID's job ends here; everything from this point is handled by the platform.
(3)
Token is checked on every request — and unlocks the citizen's own data
Every call passes through the APIM gateway which validates the token at stage ① before anything reaches a registry. Certain registries — particularly the Taxpayer Registry — require this token to be present and belong to the citizen before returning any data. No other system can use this route to access a citizen's records.
Provider-agnostic
If Sint Maarten switches eID provider, only the Entra configuration changes — no adapters, no portal code, nothing else.
⚠ Legal framework needed
A legal ordinance recognising eID for government services must be in place before go-live. Status to be confirmed.

Component 08 — Security

Security — layered protection and a phased operations model

Five layers of protection — outermost to innermost
What each layer does
Perimeter Blocks malicious traffic before it reaches the platform. DDoS protection absorbs attack volume.
Network All internal services communicate over private connections — not the public internet. Nothing is exposed unless explicitly permitted.
Application Every request is inspected by the APIM gateway's 8-stage policy pipeline. Every code change is automatically scanned for security vulnerabilities (SAST) at stage ① of the CI/CD pipeline before it can reach any environment.
Identity Every call must carry a valid login token. Citizens can only access their own data. No system can bypass this layer.
Data All secrets and credentials are stored in Key Vault — never in code or configuration files. Services access them only through their assigned identity.
Why security monitoring is phased
Automated threat response is only safe after alert thresholds have been tuned and validated. Switching it on too early creates false alarms that erode trust and block legitimate traffic. Phase 1 monitors and observes. Phase 2 activates automated response once quality is confirmed. Phase 3 hands monitoring over to GoSXM.

Component 10 — Payment Gateway

Third-party payment integration

When a citizen completes a government service that requires a fee — a licence application, a certificate issuance, a registration — the platform routes the payment through one of two integrated payment providers. The platform never handles card data or funds directly.

📌
The platform does not process payments. It initiates a payment request, passes the citizen to the provider's secure checkout, and receives a confirmation callback when payment is complete. All card data, PCI compliance, and transaction processing are the provider's responsibility.
How a payment works
(1)
Citizen completes a service that requires payment
A fee is associated with the service — for example, submitting an Economic Licence application. The platform calculates the amount and prepares a payment request.
(2)
Platform redirects to provider's secure checkout
The citizen is redirected to the chosen payment provider — CX Pay or Sentoo. Card details are entered directly on the provider's page. The platform never sees card data.
(3)
Provider sends a confirmation callback
Once payment is confirmed, the provider calls back to the platform with a signed confirmation. The platform verifies the signature, records the transaction in the Audit Log, and triggers the next step in the service flow — for example, submitting the licence application to TEATT.
(4)
Receipt stored — citizen notified
A payment receipt is written to the Audit Log. The citizen receives a confirmation on screen and optionally by email. The service continues automatically — no manual step required.

Nothing needs rebuilding.
As the Identity slide showed, Azure Entra External ID is the broker that sits between the eID provider and the rest of the platform. It translates the eID confirmation into a platform token. If Sint Maarten switches eID provider, only the Entra federation configuration is updated — one config change. No adapter, no portal, no registry connector, no pipeline is touched. The platform was designed this way deliberately.
03
The security diagram showed three phases. Why not turn on automated threat response from day one if the technology is already there?
tap to reveal →
Because tuning comes before automation.
A security system that automatically blocks or isolates services is only safe when its alert thresholds have been validated. At launch, the team does not yet know what normal traffic patterns look like for this platform. Phase 1 monitors and observes — every alert is triaged manually. Once the team has confirmed that alerts fire correctly and not on legitimate traffic, Phase 2 activates automated response. Switching automation on too early risks blocking real citizens and eroding trust in the platform before it has even proven itself.
