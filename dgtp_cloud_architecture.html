Cloud Strategy

Three Environments, One Architecture

The same LLD design runs across Dev, Staging, and Production — only the service tier and scale change. Production itself ships in three sizings (Minimal / Standard / Full) so DGTP can launch lean and scale on evidence. All edge + network is a shared service hosted once in the Production subscription, so Dev and Staging carry $0 edge.

A
Dev Sandbox
$273/mo · App Service · synthetic data

A functional-correctness and integration environment — architecturally representative of Prod so Bicep modules and app wiring are validated, at the lowest credible cost. Multi-tenant App Service (no ASE stamp), serverless data, single region. No HA, no Sentinel, no DR. Carries no citizen PII — synthetic / seeded data only.

✓All 6 registry adapters live
✓App Service P0v3 ×2 (frontend + backend)
✓Azure SQL DB serverless + Cosmos serverless
✓Service Bus + Event Grid · edge shared from Prod
✗No HA · No Sentinel · No DR
✗No zone-redundancy · LRS storage · no PII
Best for: functional + integration testing · synthetic data only

Path: promote validated builds to Staging — identical Bicep modules, only the SKUs change

B
Staging / UAT
$679/mo · warm SQL · autoscale 1→2

The last gate before Production — UAT, integration sign-off, and light HA / performance testing. Higher-end compute and data than Dev (P1v3 autoscale, warm provisioned SQL, Cosmos autoscale, Key Vault Premium) on masked / synthetic datasets. Edge is still shared from the Prod subscription, so the whole budget goes into the workload.

✓App Service P1v3 ×2 · autoscale 1→2

✓Cosmos autoscale + Key Vault Premium
✓Defender 5 plans · App Insights → central LAW

△Single region · ZRS · masked UAT data only
Best for: last gate before Production · UAT + load testing

Path: cut over to Production at the chosen sizing — dual ASE v3, edge already in place

$2,433–4,955/mo · Minimal → Standard → Full

Full ($4,955): adds Front Door Premium, APIM Standard v2, zone-redundant 4-vCore SQL, GRS + DR region. A
Plan A · Dev Sandbox · $273/mo

Build and test the whole platform — cheaply, with no citizen data.

Non-technical analogy
Think of Plan A as the workshop garage — where you build and road-test the vehicle before it ever carries a passenger. It mirrors the real thing closely, but runs on a test track with crash-test dummies, not citizens. Cheap to run, safe to break.
What Plan A includes
🚪
The front door (shared)
Reaches the portal through the shared Prod edge
Dev doesn't pay for its own front door. It reaches the portal through the same Front Door / App Gateway / APIM edge hosted once in the Production subscription. Think of it as a shared reception desk for all three environments — host-header and path rules keep Dev traffic separated from real citizen traffic. $0 edge cost is booked to Dev.
⚙️
The engine room
App Service P0v3 ×2 — all 6 registries wired
Two small App Service plans — one front-end, one back-end — run the full request path end-to-end. No high-availability and no ASE stamp: Dev only needs to prove the wiring works, not survive an outage. All six government registries (Civil, Commerce, Tax, TEATT, Permits, Land) are connected against test endpoints.
🗄️
The data safe
Azure SQL DB serverless + Cosmos cache + Audit Log
Data lives in a serverless SQL database that auto-pauses when idle — you pay only for the hours it runs. A Cosmos cache answers most requests in milliseconds instead of calling the registry every time. The 7-year audit-log mechanism and SHA-256 identity scrambling are wired exactly as in Prod — but in Dev they hold synthetic data only, never real citizens.
🔒
Security baseline
Identity + Secrets Vault + Defender
Sign-in uses the national eID directly (external OIDC/SAML — no Azure identity service to pay for). All secrets and API keys live in Key Vault, reached only by managed identity — never hardcoded. Microsoft Defender for Cloud watches the App Service, SQL and Key Vault; the shared firewall in the Prod subscription handles network filtering.
What Plan A does NOT include — and why that's fine
🛡️
No dedicated edge or firewall
Shared from the Production subscription
Dev has no Front Door, App Gateway or firewall of its own — these are shared services hosted once in Production and consumed by every environment. This is why Dev costs $273, not thousands. The one thing to confirm at design time is that host-header / path routing keeps non-prod traffic cleanly separated on the shared edge.
📊
No Sentinel (security operations centre)
Security investigations via Azure Defender alerts
Microsoft Sentinel is a full security intelligence platform — powerful, but adds significant cost and requires a trained security team to interpret its findings. Without a dedicated SOC, activating Sentinel creates noise, not insight. Plan A uses Azure Defender alerts and log queries instead — the same data, simpler tooling.
⚖️
No high-availability failover
Single instance — fine for a test environment
Dev runs one instance per tier with no failover. If it restarts, developers wait a moment — no citizen is affected, because no citizen uses Dev. Paying for high-availability in a sandbox would be spending money to protect crash-test dummies. HA arrives in Staging and is standard in Production.
When to promote Dev → Staging
✅
All 6 adapters pass integration tests against the registry endpoints
📈
App Service CPU and Cosmos RU/s stay within budget under representative load
🔐
Security scan + Defender baseline clean on the candidate build
📋
Masked UAT data ready — cut a Staging release with the same Bicep modules

B
Plan B · Staging / UAT · $679/mo

The last gate before Production — prove it under load, on safe data.

Non-technical analogy
Plan B is the dress rehearsal on the real stage — same script, same set, a full run-through with stand-ins instead of the public. You warm up the database, rehearse the failover, and prove every cue works before opening night in Production.
What Staging adds over Dev
🔁
Autoscale headroom
App Service P1v3 — scales 1→2 under load
Staging steps up to P1v3 plans — twice the cores and RAM of Dev — that autoscale from one instance to two when a load test pushes them. This is where you prove the platform holds up under realistic citizen traffic, before it ever sees a real citizen.
🗄️
Warm, load-testable database
Azure SQL DB — provisioned 2 vCore, always warm
Dev's database auto-pauses to save money; Staging's is provisioned and always warm, so load tests measure real performance with no cold-start surprises. It's the same engine that runs in Production — just sized for UAT.
🔑
Production-parity data tier
Cosmos autoscale + Key Vault Premium (HSM keys)
The Cosmos cache moves to autoscale — it bills only for the busiest hour — and Key Vault upgrades to Premium with HSM-backed keys, the same key-protection standard Production uses. Staging now mirrors Prod's data and secrets behaviour exactly.
📡
Broader threat protection
Microsoft Defender — 5 plans active
Staging widens Microsoft Defender for Cloud from three plans to five — adding Storage and Containers to App Service, SQL and Key Vault. Telemetry still flows to the central Log Analytics workspace shared by all environments, so there's no separate SIEM cost here.
What stays the same as Plan A
✅
Unchanged components
All 6 registry connections work identically. The citizen login (national eID) is unchanged. The 7-year audit log and SHA-256 identity scrambling are unchanged. The Cosmos cache, Service Bus message system, and Bicep infrastructure-as-code pipelines are identical. Only the scale and data tier change — not the architecture.
What Plan B still does NOT include
⚡
No dedicated edge — shared from Prod
Like Dev, Staging has no Front Door, App Gateway or firewall of its own; it rides the shared edge in the Production subscription. This keeps the whole $679 budget in the workload — where UAT actually exercises it — instead of in edge appliances that already exist in Prod.
🌍
No geographic disaster recovery
If the entire Azure data centre region has an outage, Plan B has no automatic failover to a second region. For Sint Maarten's usage patterns, a regional Azure outage is extremely rare and brief. The Cosmos DB and SQL databases have local backups that allow recovery within hours — not days.
🏗️
No production data, no zone spread
Staging carries only masked or synthetic UAT data — never real citizen records — so PCI-DSS and full-ISO controls stay scoped to Production. It also runs in a single zone; zone-redundancy is a Production (Full) feature, not a Staging one.
When to cut over Staging → Production
✅
UAT sign-off complete and load tests pass at the target citizen volume
📋
Production sizing chosen — Minimal (launch), Standard (HA) or Full (DR + geo)
🔐
PCI-DSS scope confirmed for e-Payment · eID degraded-mode plan signed off
Plan B adds: App Service P1v3 · autoscale 1→2 Warm SQL DB 2 vCore Cosmos autoscale Key Vault Premium (HSM) Defender 5 plans ZRS storage Dedicated edge (shared) Sentinel (central) Zone spread Geo-DR
C
Plan C · Production · $2,433–4,955/mo

The live citizen environment — sized Minimal, Standard, or Full.

Non-technical analogy
Plan C is opening day at the real venue — the doors open to all 50,000 citizens. You can open with one stage (Minimal), add a second for resilience (Standard), or run a full house with a backup venue in another city (Full). You grow the room as the crowd grows — you don't build the whole stadium on day one.
⚠️
Important for decision-makers: Production carries real citizen data, so it depends on four blocking gates being cleared first — shared-edge traffic isolation, the firewall-tier decision for PCI e-Payment, external-LLM PII / DPIA clearance, and the national-eID SLA / degraded-mode plan. These are compliance obligations, not optional polish.
What Plan C adds over Plan B
🏛️
Production compute — dual ASE v3
Frontend + Backend stamps, scaled to the chosen sizing
Production runs the dual App Service Environment topology — a dedicated, network-isolated frontend stamp and backend stamp. Minimal runs one instance each; Standard and Full run two each for automatic high-availability; Full spreads across availability zones. Same architecture throughout — you just add instances as the crowd grows.
🛡️
The shared edge lives here
Azure Firewall Standard + Front Door + App Gateway + APIM
All edge and network — Azure Firewall Standard ($912/mo, the single biggest line), Front Door, Application Gateway WAF v2 and API Management — is hosted once in Production and shared by every environment. Front Door and APIM step up to Premium / Standard v2 only in the Full sizing.
🔑
Hardware-protected encryption keys
Key Vault Premium — HSM-backed keys
Production protects encryption keys in a hardware security module (FIPS 140-2 Level 2) — keys are generated and stored in tamper-resistant hardware and can never be exported, even by Microsoft engineers. This replaces a far costlier dedicated Managed HSM, and is the key-protection standard required if PCI-DSS scope is confirmed for the e-Payment flow.
🌐
Tamper-proof 7-year archive
WORM immutable blob storage
Documents and the audit trail are written to WORM (write-once, read-many) immutable storage with a 7-year retention lock — a legal compliance requirement that cannot be downgraded. Storage redundancy grows with the sizing: LRS (Minimal) → ZRS (Standard) → GRS with a DR region (Full).
Formal commitments Plan C requires
📋
ADR — external-LLM removal
Azure OpenAI + AI Search removed · chatbot is external
The chatbot LLM and its search / RAG index are now an external, out-of-scope provider — no Azure-side AI is funded in any environment. This supersedes the previously locked AI tier and needs a formal ADR, plus PII / DPIA clearance before any citizen data reaches the external model.
📋
ADR — eID supersedes Entra (ADR-004)
National eID replaces Entra External ID · $0 CIAM
Citizens authenticate through the national eID (federated OIDC / SAML) — Entra External ID is removed, so there is no Azure identity cost. This supersedes ADR-004 and makes the eID's SLA and a degraded-mode fallback plan a launch blocker for go-live.
Items locked in Plan C — cannot be removed
🔒 WORM 7-yr storage
Write-once immutable archive. Legal compliance. Cannot downgrade.
🔒 7-yr audit log (LTR)
Append-only audit trail, 7-year retention. EA Ordinance requirement.
🔒 Azure Firewall Standard
Shared edge security boundary. PCI scope for e-Payment.
🔒 Front Door + WAF v2
Public-facing WAF on the citizen entry path. Cannot disable.

✓HA from Standard · zone-redundant at Full
✓GRS + DR region (netherlandsnorth) at Full
✓Shared edge: Firewall + WAF + APIM
✓FIPS 140-2 L2 hardware key protection
✓7-year WORM compliance storage
✓Sentinel (cost-optimized) + Defender
✓ISO 27001 · GDPR · PCI-DSS (e-Payment)

Locked Items

Components That Cannot Be Reduced Further

Seven components are locked in Production — they must remain regardless of optimisation pressure. Removing or downgrading any of them requires a formal ADR and, in most cases, regulatory or compliance sign-off.

🔒 WORM 7-year Immutable Storage

Documents and the audit archive are written to write-once, read-many (WORM) blob storage with a 7-year retention lock — required for audit compliance under the EA Ordinance and GDPR. Removing WORM or shortening retention is a compliance violation.

Production only · redundancy LRS→ZRS→GRS by sizing
🔒 7-year Audit Log (SQL LTR)

dbo.AuditLog is append-only with a 7-year long-term retention policy and SHA-256 scrambled citizen identifiers. The retention window and append-only behaviour are legal requirements that cannot be relaxed.

All environments wire it · only Production holds real records
🔒 Azure Firewall Standard (shared)

One shared Azure Firewall Standard ($912/mo) is the network security boundary for every environment. It is in scope for the PCI-DSS e-Payment assessment, so its tier and rules cannot be reduced without a firewall-tier ADR and security sign-off.

Hosted in Prod sub · shared by Dev / Staging / Production
🔒 Front Door + App Gateway WAF v2

The public citizen entry path runs through Front Door and an Application Gateway WAF v2 with the OWASP rule set. Disabling the WAF on a public government portal is not permitted. Front Door steps up to Premium in the Full sizing.

Production edge · shared by all environments
🔒 Key Vault HSM-backed Keys

Key Vault Premium provides FIPS 140-2 Level 2 HSM-backed keys for the e-Payment and citizen-data flows. If PCI-DSS scope is confirmed, the HSM key-protection standard is mandatory and cannot drop to software-protected keys.

Staging + Prod Standard/Full · Dev / Prod-Minimal use KV Standard
🔒 Defender for Cloud Baseline

Security baseline per ADR-009: Defender for App Service, SQL, Storage and Key Vault. Production runs the full plan set; it cannot be downgraded without an ADR-009 revision and security-team sign-off.

Dev 3 plans · Staging 5 plans · Production full
🔒 National eID Degraded-Mode Plan

Citizens sign in only through the national eID. Because there is no fallback Azure identity service, a documented eID SLA and a degraded-mode plan (what happens when eID is unavailable) is a launch blocker that cannot be skipped.

External dependency · supersedes ADR-004 (Entra removed)
⚠ ADRs Required Before Production

ADR — external-LLM removal: Azure OpenAI + AI Search removed; chatbot LLM + search are external. Supersedes the locked AI tier; needs PII / DPIA clearance.

ADR — eID supersedes Entra (ADR-004): national eID replaces Entra External ID. ADR-011: shared-edge traffic isolation across Dev / Staging / Prod.

Recommendation

Deploy Dev → Staging → Production.
Size Production on signal — not on schedule.

Sint Maarten has 50,000 citizens. The full three-environment estimate lands at $3,385–5,907/mo — a fraction of the earlier baseline — by sharing the edge, removing Azure-side AI, and using the national eID. Stand up Dev and Staging first, then launch Production at the Minimal sizing and grow to Standard (HA) and Full (DR) only as real load justifies it.

Phase 1 · Dev + Staging

Stand up Dev ($273) and Staging ($679) first. All 6 adapters live on App Service, Azure SQL DB, Cosmos and Service Bus. Validate the full request path and the shared edge on synthetic, then masked, data. Bicep modules are identical to Production.

Phase 2 · Production — Minimal

Launch Production at the Minimal sizing ($2,433): dual ASE v3 single-instance, serverless SQL, shared Firewall Standard + Front Door + APIM. Go live for 50k citizens, then collect real load — CPU, RU/s, request rates — for 3 months.

Phase 3 · Production — Standard → Full

Grow only when the data justifies it: move to Standard ($3,419) for 2-instance HA and warm SQL, then to Full ($4,955) for Front Door Premium, APIM Standard v2, zone-redundant SQL and GRS + a DR region. Each step is a parameter change — no redesign.
