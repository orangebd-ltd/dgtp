Design Principles

Five principles behind every decision

These are not aspirational statements. Each principle has a direct, verifiable expression in the architecture.

Marcus · sole proprietor · Sint Maarten
•Every design decision was made with Marcus in mind. •He should only need to provide his information once. •His data should never leave Sint Maarten. He should only ever see his own records. •And the system should keep working even if one registry has a fault. tap for next →
01

Open, Standards-Based Integration

Every registry uses an open, industry-OData · Tax CRM. If a vendor changes, only the adapter configuration changes. No lock-in by design.

02

Authoritative Source · Loosely Coupled

Each registry remains the owner of its own data — nothing is migrated or duplicated permanently. Independent adapters mean one registry failure cannot bring down the others.

03

Security, Identity & Privacy by Design

Eight security gates on every request. eID login required. Citizens access only their own data. Audit IDs scrambled. Compliant with Sint Maarten government data requirements.

04

Resilience & Once-Only by Default

Citizens provide information once — the platform fetches it automatically. The platform guarantees zero data loss. Shared data records refresh promptly. Audit log retained 7 years.

05

Observable & Reproducible Infrastructure

Every log line carries a unique tracking ID. All infrastructure is code — reproducible from a single pipeline run. Security scanned on every commit. Alert rules fire automatically on threshold breach.

Security Architecture

Every citizen request passes 8 security gates

Before any registry data reaches a citizen, the platform runs every request through eight sequential checks. Any check can stop the request entirely. Only requests that pass all eight reach the registries.

Marcus · sole proprietor · Sint Maarten
•Every time Marcus taps a button, his request passes through 8 security checks. •If he is not logged in, it is blocked at gate one — the registry is never contacted. •He never sees this — but it protects him every single time. tap for next →
Five layers of protection — outermost to innermost
🔐
Any gate can block a request
If a citizen is not logged in, the request never proceeds past gate one — the registry is never contacted.
🧾
Every request is permanently traceable
Gate ⑥ stamps a unique ID on every request — linking it to every log, every registry call, and the full audit trail.
🙋
Citizens can only access their own data
The login token identifies who is making the request. No government system or third party can use this route to access a citizen's records.
🛡️
Security is phased
Phase 1 monitors and observes — automated blocking is only activated in Phase 2 once alert thresholds are validated against real traffic patterns.
Identity & Access

How Sint Maarten citizens log in securely

Citizens use their national eID — the same government-issued digital identity — to access all six registries through one login. The platform never handles passwords or credentials. That responsibility stays entirely with the national eID system.

Marcus · sole proprietor · Sint Maarten
•Marcus logs in with his national eID — the same government-issued identity he already has. •He taps Login, confirms his identity, and receives a secure session pass. •He never creates a new account, never sets a new password. •And can only access his own records. tap for next →
What happens when a citizen logs in
(1)
Citizen taps "Login" on the portal
The portal redirects the citizen to the government's eID login page. The platform itself never sees their password or any personal credentials.
(2)
National eID confirms who the citizen is
The government's eID system authenticates the citizen and sends a secure confirmation back. This is the same protocol used by major identity systems worldwide.
(3)
The platform's identity layer converts the confirmation
The platform receives the eID's confirmation and converts it into a short-lived access pass for the citizen's session. The eID's job ends here — everything from this point is handled by the platform.
(4)
An access pass is issued to the citizen's session
A short-lived, secure pass is issued. It carries no personal data — only a scrambled identifier and what the citizen is permitted to see. It expires automatically.
(5)
The pass is checked on every action — own data only
Every subsequent action by the citizen is verified against their pass. Citizens can only access their own data. No government system or third party can use this route to access a citizen's records.
Future-proof
If Sint Maarten changes its eID provider in the future, only one configuration changes. The portal, registries, and all other components remain untouched.
🔒 Citizens own their data
The access pass identifies the citizen. No other system can use this authentication route to retrieve a citizen's records without their explicit login.
